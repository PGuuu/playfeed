# PlayFeed 遊戲投稿規範 v1

> 這份文件是給「想投稿遊戲到 PlayFeed 的人（或他們的 AI）」看的完整規格。
> 只要照這份文件寫出來的遊戲，就能在 PlayFeed 上跑、上排行榜、被別人改造（remix）。
>
> 給 AI 的話：把這整份當成規格書。你的產出必須是**一個 `.js` 檔**，
> 內容是一個符合下方「遊戲物件」格式的物件，且遵守所有「硬性規則」。
> 寫完請自己對照最後的「送審檢查表」逐項確認。
>
> **這份文件只規定如何在 PlayFeed 執行，不規定遊戲應該怎麼設計。**
> 請忠實發展使用者的點子；經典玩法與新玩法都可以。不要把欄位、骨架或 API 名稱當成玩法範例。

---

## 0. 三十秒摘要（TL;DR）

- 交付物：**單一 `.js` 檔**，把一個遊戲物件註冊進平台。
- 畫面：固定邏輯尺寸 **400 × 700**（直式），全部畫在提供的 canvas 上。
- 操作：**只能用「點一下 / 左右滑 / 按住放開」**。**嚴禁任何上下（垂直）滑動手勢**——上下滑是平台用來切換遊戲的，你搶不到、也不准搶。
- 自足：**不可連網、不可讀寫儲存、不可碰 DOM、不可載入外部圖片/音檔/字型**。畫面用 canvas 畫，聲音用內建 `beep()`。
- `duration` 填寫 20～60 秒的預估單局時間。
- 結束一定要呼叫 `over(分數)`；分數變動呼叫 `setScore(分數)`。
- 檔案大小上限 **200 KB**，UTF-8 純文字。

---

## 1. 硬性規則（違反就退件，無例外）

### 1.1 嚴禁垂直（上下）滑動手勢 ★最重要
上下滑是 feed 的專屬手勢（切換上一個／下一個遊戲）。技術上平台會把畫布設成
`touch-action: none` 並自行偵測垂直滑動，**你的遊戲根本收不到垂直滑動**——玩家一往上／下滑，
畫面就會被平台捲走。所以：

- ✅ 允許的操作：**點一下（tap）**、**左右方向的拖曳／滑動**、**按住不放 → 放開**。
- ❌ 禁止：往上滑、往下滑、需要「畫圈」或「大幅斜向」的手勢、任何需要玩家做出上下方向動作才能玩的機制。
- 如果玩法需要方向操作，請改用點按區域、左右選擇或水平拖曳來實作。

### 1.2 單一檔案、單一遊戲、指定格式
一個 `.js` 檔 = 一款遊戲。檔案內容就是把遊戲物件註冊進平台：

```js
window.GAMES = (window.GAMES || []).concat([
  {
    apiVersion: 1, gameVersion: '1.0.0',
    id: 'your-id-suggestion', title: '遊戲名稱',
    description: '一句話介紹', author: '@草稿作者',
    tip: '一句話講完怎麼玩', bg: '#123456',
    tags: [], controls: ['tap'], duration: 45,
    score: { label: '分數', order: 'higher' },
    create(env) {
      // ...
      return { start() {…}, stop() {…}, input(t, x, y) {…} };
    }
  }
]);
```

### 1.3 完全自足，禁止對外
你的程式碼**只能**用下方「§3 允許的 API」。以下一律禁止（送審會被自動擋下）：

- 連網：`fetch` / `XMLHttpRequest` / `WebSocket` / `EventSource` / `navigator.sendBeacon` / 動態 `import()`。
- 儲存：`localStorage` / `sessionStorage` / `indexedDB` / `document.cookie`。
- DOM／瀏覽器：`document` / `window.*`（唯一例外是 `window.GAMES` 的註冊）/ `location` / `navigator.*` / 建立或操作任何 HTML 元素 / `addEventListener` / `window.open`。
- 動態程式碼：`eval` / `new Function()` / `Function(...)`。
- 外部資源：用網址載入圖片、音檔、字型、影片。**畫面全部自己用 `ctx` 畫，聲音只能用 `beep()`。**
- 裝置：相機、麥克風、定位、感測器、剪貼簿。

> 為什麼這麼嚴？因為 PlayFeed 會在同一個頁面上跑很多人上傳的遊戲，
> 上面那些能力會被拿來偷資料、洗版、或破壞別人的遊戲。詳見 §7 平台如何強制執行。

### 1.4 一定要能乾淨結束與停止
- 遊戲結束（角色死亡／時間到）**必須呼叫 `env.over(finalScore)`**，否則玩家卡死、也上不了排行榜。
- 回傳的 `stop()` **必須**把所有 `requestAnimationFrame` 取消、所有 `setTimeout/setInterval` 清掉、並停止繼續繪製。
  玩家隨時可能滑走，沒停乾淨會造成背景耗電、甚至畫到別人的遊戲上。

### 1.5 座標系統固定 400 × 700
所有座標、尺寸都以邏輯畫布 **400（寬）× 700（高）** 為準（用 `env.W`、`env.H`）。
不要用實際像素、不要假設螢幕大小；平台會自動把 400×700 縮放到各種手機。

## 2. 遊戲物件（Game Object）規格

```ts
interface Game {
  apiVersion: 1;       // Runtime API 版本
  gameVersion: string; // 遊戲內容版本，如 '1.0.0'
  id: string;          // slug 建議；正式 ID 與重複處理由平台決定
  title: string;       // 顯示名稱
  description: string; // 一句話介紹
  author: string;      // 僅供草稿預覽；正式作者由登入帳號覆蓋
  tip: string;         // 一行操作說明
  bg: string;          // 背景色（hex），會用在貼文底色與載入時的過場
  tags: string[];      // 分類與推薦
  controls: ('tap'|'horizontal'|'hold')[]; // 不可包含垂直操作
  duration: number;    // 預估秒數，20～60
  score: { label: string; order: 'higher'|'lower' };
  remixSlots?: Slot[]; // 選配：宣告可被別人換圖的元素（見 §5）
  create(env: Env): GameInstance;
}

interface GameInstance {
  start(): void;                         // 重設狀態並開始（通常 = reset() + requestAnimationFrame(loop)）
  stop(): void;                          // 停止並清乾淨（cancelAnimationFrame + 清 timers + alive=false）
  input(type: 'down'|'move'|'up'|'cancel', x: number, y: number): void;
}
```

- `create(env)` 只會被呼叫來建立一個實例；真正開始玩是呼叫回傳的 `start()`。
- 同一個實例可能被 `start()` / `stop()` 多次（玩家一直重玩）。`start()` 要能把狀態完整重設。
- 可以先宣告 `start`、`stop`、`input`，最後寫 `return { start, stop, input }`。`create(env)` 內的輔助函式可以正常回傳座標、角色等資料物件；平台只檢查 `create(env)` 自己直接回傳的 GameInstance，不會把巢狀輔助函式的 `return { ... }` 當成 GameInstance。

## 3. 允許使用的 API：`env`

`create(env)` 收到的 `env` 是你**唯一**能碰的平台介面。除了 `env` 和標準的純 JS
（`Math`、`Array`、`Date.now()`、`requestAnimationFrame`/`cancelAnimationFrame`、
`setTimeout`/`setInterval` 及對應的清除函式）之外，什麼都不要碰。

| 成員 | 型別 | 說明 |
|---|---|---|
| `env.W` | `number` | 畫布寬 = 400 |
| `env.H` | `number` | 畫布高 = 700 |
| `env.ctx` | `CanvasRenderingContext2D` | 你的畫布。只用 2D 繪圖方法，別去改 canvas 元素本身 |
| `env.setScore(n)` | `(n:number)=>void` | 更新畫面上顯示的分數。分數變動時呼叫 |
| `env.over(score)` | `(score:number)=>void` | 結束這一局，帶入最終分數。上排行榜、顯示結算畫面都靠它 |
| `env.beep(f1,f2,dur,vol,type)` | 函式 | 產生音效。`f1→f2` 是頻率滑音（Hz），`dur` 秒，`vol` 0~1，`type` 為 `'sine'\|'square'\|'sawtooth'\|'triangle'` |
| `env.getSprite(key)` | `()=>Image\|null` | 若某可換圖元素被玩家換過，回傳該圖；否則 `null`。見 §5 |
| `env.sprite(key,cx,cy,size,flip?)` | `()=>boolean` | 便利函式：若該元素有換圖，就置中等比畫出並回傳 `true`；否則回傳 `false`（你再畫預設外觀）。見 §5 |

> 相容性：目前平台也會提供全域 `W`、`H`、`beep`，但**請一律用 `env.` 版本**，未來沙盒化後只有 `env` 保證存在。

## 4. 輸入模型：`input(type, x, y)`

平台會把玩家的觸控／滑鼠翻譯成四種事件丟給你：

| `type` | 何時 | `x,y` |
|---|---|---|
| `'down'` | 手指按下／滑鼠按下 | 按下的位置（邏輯座標，0≤x≤400、0≤y≤700） |
| `'move'` | 按住移動（含水平拖曳） | 目前位置 |
| `'up'` | 玩家真的放開；可以執行發射等放開行為 | 最後位置 |
| `'cancel'` | 系統中止（例如玩家開始上下滑切換遊戲） | 最後位置 |

規則與注意事項：

- 只准用這四種事件做出「點 / 水平拖 / 按住放開」。**不要用 x,y 的垂直變化當作操作訊號**（那是 feed 的）。
- `'cancel'` 只解除按住／拖曳狀態，不可發射、計分或觸發其他玩家行為。
- 桌機鍵盤：平台會把方向鍵/空白鍵轉成對應的 `input('down', …)`（左＝W×0.25、右＝W×0.75、上/空白＝W/2）。你只要處理好 `input` 即可，不用自己聽鍵盤。
- 你的遊戲**不可以**自己 `addEventListener`。所有輸入只從 `input()` 來。

## 5. 支援 Remix（可換圖，完全選配）

PlayFeed 的特色是「別人可以上傳圖片，把遊戲裡的元素換成自己的」。
只有在遊戲中確實有適合替換的視覺元素時，才宣告 `remixSlots`，並在畫圖時用 `env.sprite()`：

```js
remixSlots: [
  // key：程式用的代號；label/hint：面板上給玩家看的；default：沒換圖時顯示的 emoji；
  // shape：輪廓提示，讓玩家知道圖會被縮放成什麼形狀 'free'|'circle'|'wide'|'tall'
  { key: 'element', label: '可替換元素', hint: '顯示在遊戲中的元素', default: '●', shape: 'free' },
],
```

畫圖時：

```js
// 有換圖就畫圖並回傳 true；沒有就自己畫預設外觀
if (!env.sprite('element', x, y, 74)) {
  // …用 ctx 畫原本外觀…
}
```

- 若要自己控制繪製，也可以 `const im = env.getSprite('element')`（回傳 `Image` 或 `null`），
  再自行用 `ctx.drawImage` 繪製。
- 沒有合適元素時直接省略 `remixSlots`，不要為了符合格式硬加。

## 6. 創作自由與執行品質

- PlayFeed 不限制題材、玩法、類型、節奏、難度曲線或是否採用經典機制。
- 規格中的 metadata、API 和程式骨架都不是玩法範例。
- `tip` 應清楚說明實際操作，但平台不規定遊戲必須採用哪一種設計公式。
- 平台會用合法的隨機輸入產生 Feed 預覽；遊戲不必迎合亂玩效果，但收到這些輸入時不可卡死或報錯。
- 每幀避免過重運算或大量建立物件，讓中階手機也能順暢執行。
- 內容須符合 PG-13，不得包含違法、仇恨、色情、暴力露骨、騷擾或侵權內容。

## 7. 平台如何強制執行（給平台方 / 也讓投稿者了解界線）

> 這一節說明「限制」實際上是怎麼被擋下來的。投稿者了解即可；平台方在開放公開投稿前**必須**先建好這一層。

1. **檔案檢查（自動）**：投稿的 `.js` 會先跑 `tools/validate-game.mjs`（見同層檔案）做靜態檢查——副檔名、大小、必要欄位、以及 §1.3 的禁用字串（`fetch`、`document`、`eval`…）。任何一項失敗直接退件。這是**第一道篩**，不是安全邊界。
2. **沙盒執行（自動，安全邊界）**：通過靜態檢查的遊戲，**不會**用 `<script src>` 直接塞進主頁面（那樣它就能偷 Supabase/Phuze 金鑰、洗資料庫、竄改別人的遊戲）。正式上線時每款遊戲跑在
   `<iframe sandbox="allow-scripts">`（不同源、無 `allow-same-origin`）裡，
   搭配 CSP `default-src 'none'` 封死一切網路。iframe 內有自己的 canvas 與 `ctx`；主頁只透過 `postMessage` 傳入輸入，接收分數、結束與截圖。遊戲拿不到主頁的任何東西。
3. **人工審核 + 檢舉**：自動關卡擋掉技術性違規，但「無聊、抄襲、內容不當」需要人看。每款遊戲上線前人工過一遍，上線後保留「檢舉」按鈕與下架機制。
4. **配額與速率**：單一投稿者的上傳數量／頻率設上限，避免洗版。

> 現況：`pack1`～`pack5` 是平台官方自己寫的可信遊戲，用 `<script src>` 載入沒問題。
> **在沙盒（第 2 點）做好之前，不要開放陌生人直接上架**——只用「投稿 → 你本人審過 → 手動加進 packN」的流程過渡。

## 8. 送審檢查表（投稿前自己逐項打勾）

**格式**
- [ ] 單一 `.js` 檔、UTF-8、≤ 200 KB。
- [ ] 用 `window.GAMES = (window.GAMES || []).concat([{…}])` 註冊，且只有一款遊戲。
- [ ] 有 `apiVersion` / `gameVersion` / `id` / `title` / `description` / `tip` / `bg` / `tags` / `controls` / `duration` / `score` / `create`。
- [ ] `create(env)` 回傳的物件有 `start()` / `stop()` / `input(t,x,y)`。

**行為**
- [ ] 完全沒有上下（垂直）滑動的操作機制。
- [ ] 只用點按 / 水平拖曳 / 按住放開。
- [ ] 結束會呼叫 `env.over(score)`；分數變動呼叫 `env.setScore(n)`。
- [ ] `stop()` 有 `cancelAnimationFrame` 並清掉所有 timer，停止繪製。
- [ ] `'cancel'` 只安全解除狀態，不觸發放開行為。

**自足**
- [ ] 沒有 `fetch`/`XMLHttpRequest`/`WebSocket`/`import()`。
- [ ] 沒有 `localStorage`/`sessionStorage`/`cookie`/`indexedDB`。
- [ ] 沒有 `document`/`window.*`(除 `window.GAMES`)/`location`/`navigator.*`/`addEventListener`。
- [ ] 沒有 `eval`/`new Function`。
- [ ] 沒有用網址載入圖片/音檔/字型；畫面全用 `ctx`，聲音只用 `beep()`。

**執行品質**
- [ ] `tip` 能正確說明操作。
- [ ] `duration` 是 20～60 的整數。
- [ ] 收到合法的隨機輸入也不會卡死或報錯。
- [ ] 只有在適合遊戲時才宣告 `remixSlots`。

**工具**：本機可跑 `node tools/validate-game.mjs 你的遊戲.js` 先自動檢查一遍。

## 9. 關於程式骨架

本文件 §1.2 的程式碼只示範必要資料結構與生命週期，不是一款遊戲，也不暗示任何玩法。
PlayFeed 不在主要創作規格中提供完整遊戲範例，避免範例影響創作者或 AI 的設計方向。
